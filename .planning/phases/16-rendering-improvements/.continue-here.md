- **Location:** `src/main/kotlin/geo/render/PolygonRenderer.kt:127-131`
- **Issue:** Contours created without winding normalization
- **Impact:** `compound { difference {} }` requires consistent winding for boolean subtraction
- **Evidence:** Working code paths (MultiRenderer, OptimizedFeature) explicitly use `.clockwise`/`.counterClockwise`

<context>

**The debugging journey:**

Started with visual verification of gap closure plan 16-04. User reported all visual tests failed despite code appearing correct. Ran 8 UAT tests systematically:

1. **Ocean test** - Immediate crash (lat 90°)
2. **Hole tests** - Solid polygons, no cutouts
3. **Code verification** - Everything looks right but doesn't work

**Breakthrough moment:**

Debug agent noticed working code paths use explicit winding (`.clockwise`/`.counterClockwise`) but `writePolygonWithHoles()` creates raw contours. GeoJSON has CCW exterior/CW holes, OPENRNDR needs consistent winding for `compound { difference {} }`.

**Why this matters:**

The `compound { difference {} }` API performs boolean shape subtraction, not winding-based fill. Without normalized winding, the subtraction produces incorrect results even though all the data is correct.

**Key insight from user:**

"QGIS fills polygons using scanline rasterizer (fills between edge crossings per row) while OPENRNDR traces contour path as vector and fills enclosed shape. For antimeridian-wrapping coordinates, vector path zigzags producing different fill artifacts."

This explains why antimeridian splitting works in QGIS but not OPENRNDR - different rendering models require different normalization strategies.
