# Phase 4.1: CRS-Aware GeoSource with Auto-Reprojection - Research

**Researched:** 2026-02-22
**Domain:** CRS (Coordinate Reference System) transformations using proj4j, Kotlin sequences
**Confidence:** HIGH

## Summary

This research investigates the technical implementation details for adding CRS-awareness to GeoSource with automatic coordinate transformations. The phase aims to eliminate the ~50 lines of manual CRS handling boilerplate seen in examples like LiveRendering.kt by making GeoSource responsible for presenting coordinates in a canonical CRS (WGS84).

The implementation uses proj4j library (already in codebase at version 1.4.1) for CRS transformations, with proj4j-epsg providing EPSG code support. The design follows Kotlin's sealed class pattern for polymorphic geometry transformations and leverages lazy sequences for memory-efficient processing (with eager materialization for render loops).

**Primary recommendation:** Use proj4j's CoordinateTransformFactory and CRSFactory pattern with in-place ProjCoordinate transformations, extend Geometry with polymorphic `transform()` method, and provide both lazy `autoTransformTo()` and eager `materialize()` methods on GeoSource.

### Pattern 1: CRSTransformer Wrapper Pattern
**What:** A lightweight wrapper around proj4j's CoordinateTransform that manages CRS creation and transforms
**When to use:** When you need to perform many coordinate transforms between two CRS pairs
**Example:**
```kotlin
class CRSTransformer(
    sourceCRS: String,
    targetCRS: String
) {
    private val crsFactory = CRSFactory()
    private val transformFactory = CoordinateTransformFactory()

    private val sourceCrs = crsFactory.createFromName(sourceCRS.lowercase())
        ?: throw CRSTransformationException(
            "Unsupported source CRS: $sourceCRS. " +
            "Ensure proj4j-epsg dependency is included."
        )
    private val targetCrs = crsFactory.createFromName(targetCRS.lowercase())
        ?: throw CRSTransformationException(
            "Unsupported target CRS: $targetCRS. " +
            "Ensure proj4j-epsg dependency is included."
        )

    private val transform = transformFactory.createTransform(sourceCrs, targetCrs)

    fun transform(x: Double, y: Double): Vector2 {
        val src = ProjCoordinate(x, y)
        val dst = ProjCoordinate()
        transform.transform(src, dst)
        return Vector2(dst.x, dst.y)
    }
}
```

### Pattern 2: Sealed Class Polymorphic Transform
**What:** Add `transform()` function to sealed Geometry classes with type-specific implementations
**When to use:** When you need to transform all geometry types in a type-safe way
**Example:**
```kotlin
sealed class Geometry {
    abstract val boundingBox: Bounds
}

fun Geometry.transform(transformer: CRSTransformer): Geometry = when (this) {
    is Point -> Point(transformer.transform(x, y))
    is LineString -> LineString(points.map { TransformerPoint(transformer, it) })
    is Polygon -> Polygon(
        exterior.map { transformer.transform(it) },
        interiors.map { ring -> ring.map { transformer.transform(it) } }
    )
    // ... etc
}
```

**Critical:** Use sealed class pattern with `when` expression. The compiler ensures all geometry types are handled.

### Pattern 3: Lazy-to-Eager Sequence Adapter
**What:** Provide lazy `features: Sequence<Feature>` and eager `materialize(): GeoSource`
**When to use:** Default to lazy for memory efficiency, eager for render loops
**Example:**
```kotlin
abstract class GeoSource(val crs: String = "EPSG:4326") {
    abstract val features: Sequence<Feature>

    fun autoTransformTo(targetCRS: String): GeoSource {
        if (crs == targetCRS) return this  // Identity check, returns same instance

        val transformer = CRSTransformer(crs, targetCRS)

        return object : GeoSource(targetCRS) {
            override val features: Sequence<Feature> =
                this@GeoSource.features.map { feature ->
                    Feature(
                        geometry = feature.geometry.transform(transformer),
                        properties = feature.properties
                    )
                }
        }
    }

    fun materialize(): GeoSource {
        val materializedFeatures = features.toList()

        return object : GeoSource(crs) {
            override val features: Sequence<Feature> = materializedFeatures.asSequence()
        }
    }
}
```

**Performance note:** Lazy sequences re-run CRS transforms on every frame. Eager `materialize()` caches upfront but uses more memory. Choose based on use case (single-pass vs render loops).

### Anti-Patterns to Avoid
- **Creating CRSTransformer per coordinate:** Always create once and reuse. Creating in loop causes O(n^2) overhead.
- **Calling `toList()` unnecessarily:** Only use `materialize()` for render loops; otherwise keep lazy.
- **Swapping x/y in ProjCoordinate:** Longitude first, latitude second: `ProjCoordinate(lon, lat)` NOT `ProjCoordinate(lat, lon)`.
- **Assuming GeoJSON has CRS property:** RFC 7946 removed it. Default to "EPSG:4326".

## Don't Hand-Roll

Problems that look simple but have existing solutions:

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| CRS transformation math | Manual proj4j formulas | proj4j CoordinateTransform | Handle datum shifts, grid corrections, edge cases |
| Coordinate conversion | Manual lat/lon to easting/northing | proj4j transformation | Handles all CRS types, not just BNG |
| CRS validation | Custom EPSG lookup | CRSFactory.createFromName() | Checks EPSG registry, provides clear error |
| Optimization | Custom coordinate caching | Sequence.map() lazy evaluation | Built-in lazy processing, simpler code |

**Key insight:** proj4j handles all the geodetic complexity (datum transformations, map projections, coordinate systems). Don't re-implement PROJ.4 math yourself.

## Common Pitfalls

### Pitfall 1: Missing proj4j-epsg Dependency
**What goes wrong:** `CRSFactory.createFromName("EPSG:27700")` throws `IllegalStateException: Unable to access CRS file: proj4/nad/epsg`
**Why it happens:** proj4j-core doesn't include EPSG database (moved to separate EPSG module in proj4j 1.2.2+ for licensing reasons)
**How to avoid:** Ensure both `proj4j` and `proj4j-epsg` dependencies at the same version:
```kotlin
implementation("org.locationtech.proj4j:proj4j:1.4.1")
implementation("org.locationtech.proj4j:proj4j-epsg:1.4.1")
```
**Warning signs:** Error message "Unable to access CRS file" when calling `createFromName()`

### Pitfall 2: CRSFactory.createFromName() Returns null
**What goes wrong:** `createFromName()` returns null causing NullPointerException when calling `createTransform()`
**Why it happens:** Unknown EPSG code or missing proj4j-epsg dependency. Case sensitivity issues (EPSG vs epsg)
**How to avoid:**
```kotlin
private val sourceCrs = crsFactory.createFromName(sourceCRS.lowercase())
    ?: throw CRSTransformationException("Unsupported CRS: $sourceCRS")
```
**Warning signs:** NullPointerException in `createTransform()`

### Pitfall 3: Lat/Lon Order Confusion
**What goes wrong:** Coordinates swapped, causing locations to be off by thousands of miles
**Why it happens:** ProjCoordinate order is `(longitude, latitude)`, but GIS conventions vary
**How to avoid:** Always use `(lon, lat)` order when creating ProjCoordinate:
```kotlin
// CORRECT: Longitude first
val point = ProjCoordinate(longitude, latitude)
```
**Warning signs:** Features appear in wrong location (e.g., Africa instead of London)

### Pitfall 4: Forgetting Identity Check
**What goes wrong:** Unnecessary transformation overhead when source and target CRS match
**Why it happens:** Not performing `if (crs == targetCRS) return this` check
**How to avoid:**
```kotlin
fun autoTransformTo(targetCRS: String): GeoSource {
    if (crs == targetCRS) return this  // Early return, same instance
    // ... rest of transformation
}
```
**Warning signs:** Same CRS transformation adds measurable overhead for large datasets

### Pitfall 5: GeoJSON CRS Misunderstanding
**What goes wrong:** Trying to read `crs` property from GeoJSON, results in null
**Why it happens:** RFC 7946 (2016) removed `crs` member from GeoJSON spec. All coordinates assumed WGS84
**How to avoid:**
```kotlin
// GeoJSON always uses WGS84 (EPSG:4326)
class GeoJSONSource(
    override val features: Sequence<Feature>,
    crs: String = "EPSG:4326"  // Always WGS84
) : GeoSource(crs)
```
**Note:** Older GeoJSON 2008 spec had `crs` member but it's obsolete and unsupported

### Pitfall 6: Creating CRSTransformer in Loop
**What goes wrong:** Severe performance degradation for large datasets
**Why it happens:** Each coordinate transformation creates new CRSFactory and CoordinateTransform instances
**How to avoid:** Create CRSTransformer once in `autoTransformTo()` and reuse:
```kotlin
fun autoTransformTo(targetCRS: String): GeoSource {
    val transformer = CRSTransformer(crs, targetCRS)  // Created once

    return object : GeoSource(targetCRS) {
        override val features: Sequence<Feature> =
            this@GeoSource.features.map { feature ->
                // transformer reused for all features
                feature.geometry.transform(transformer)
            }
    }
}
```

## Code Examples

### CRSTransformer Implementation
```kotlin
// Source: proj4j projection usage pattern from ProjectionBNG.kt
import org.locationtech.proj4j.CRSFactory
import org.locationtech.proj4j.CoordinateTransformFactory
import org.locationtech.proj4j.ProjCoordinate
import org.openrndr.math.Vector2

class CRSTransformer(
    sourceCRS: String,
    targetCRS: String
) {
    private val crsFactory = CRSFactory()
    private val transformFactory = CoordinateTransformFactory()

    private val sourceCrs = crsFactory.createFromName(sourceCRS.lowercase())
        ?: throw CRSTransformationException(
            "Unsupported source CRS: $sourceCRS. " +
            "Ensure proj4j-epsg dependency is included."
        )

    private val targetCrs = crsFactory.createFromName(targetCRS.lowercase())
        ?: throw CRSTransformationException(
            "Unsupported target CRS: $targetCRS. " +
            "Ensure proj4j-epsg dependency is included."
        )

    private val transform = transformFactory.createTransform(sourceCrs, targetCrs)

    /**
     * Transforms a single coordinate pair.
     *
     * @param x X coordinate (longitude in degrees or easting in meters)
     * @param y Y coordinate (latitude in degrees or northing in meters)
     * @return Transformed coordinates as Vector2
     */
    fun transform(x: Double, y: Double): Vector2 {
        val src = ProjCoordinate(x, y)
        val dst = ProjCoordinate()
        transform.transform(src, dst)
        return Vector2(dst.x, dst.y)
    }
}

class CRSTransformationException(message: String) : Exception(message)
```

### Geometry.transform() Extension
```kotlin
// Source: Geometry.kt sealed class structure
import org.openrndr.math.Vector2

/**
 * Transforms geometry coordinates using the given CRS transformer.
 * Uses sealed class pattern for type-safe, exhaustive transformation.
 */
fun Geometry.transform(transformer: CRSTransformer): Geometry = when (this) {
    is Point -> Point(transformer.transform(x, y))

    is LineString -> LineString(
        points.map { transformer.transform(it.x, it.y) }
    )

    is Polygon -> Polygon(
        exterior.map { transformer.transform(it.x, it.y) },
        interiors.map { ring ->
            ring.map { transformer.transform(it.x, it.y) }
        }
    )

    is MultiPoint -> MultiPoint(
        points.map { point ->
            Point(transformer.transform(point.x, point.y))
        }
    )

    is MultiLineString -> MultiLineString(
        lineStrings.map { line ->
            LineString(line.points.map { transformer.transform(it.x, it.y) })
        }
    )

    is MultiPolygon -> MultiPolygon(
        polygons.map { poly ->
            Polygon(
                poly.exterior.map { transformer.transform(it.x, it.y) },
                poly.interiors.map { ring ->
                    ring.map { transformer.transform(it.x, it.y) }
                }
            )
        }
    )
}
```

### CRS Detection in GeoPackage
```kotlin
// Source: GeoPackage.kt existing CRS detection pattern
import mil.nga.geopackage.GeoPackageManager
import mil.nga.geopackage.features.user.FeatureDao

/**
 * Determines the CRS from the first feature table, defaulting to WGS84.
 * Uses FeatureDao.getSrs() to get SRS ID.
 */
private fun determineCRS(
    geoPackage: mil.nga.geopackage.GeoPackage,
    featureTables: List<String>
): String {
    return try {
        if (featureTables.isNotEmpty()) {
            val dao = geoPackage.getFeatureDao(featureTables[0])
            val srs = dao.getSrs()
            "EPSG:${srs.getSrsId()}"
        } else {
            "EPSG:4326"
        }
    } catch (e: Exception) {
        "EPSG:4326" // WGS84 default
    }
}
```

### GeoSource Extensions (CRSExtensions.kt)
```kotlin
// File: src/main/kotlin/geo/projection/CRSExtensions.kt
package geo.projection

import geo.*

/**
 * Convenience extension to transform source to WGS84 (EPSG:4326).
 * Recommended first step after loading data, since all GeoProjection
 * implementations expect WGS84 coordinates.
 *
 * Example:
 * ```kotlin
 * val data = GeoPackage.load("file.gpkg").toWGS84()
 * ```
 */
fun GeoSource.toWGS84(): GeoSource = autoTransformTo("EPSG:4326")

/**
 * Convenience extension to transform source to Web Mercator (EPSG:3857).
 * Useful for tiled map rendering.
 *
 * Example:
 * ```kotlin
 * val data = GeoJSON.load("file.geojson").toWebMercator()
 * ```
 */
fun GeoSource.toWebMercator(): GeoSource = autoTransformTo("EPSG:3857")

/**
 * Materializes lazy sequences into in-memory lists.
 * Use this for render loops where features are accessed multiple times.
 * Tradeoff: Lazy = per-frame CRS transform cost; eager = upfront cost + memory.
 *
 * Example:
 * ```kotlin
 * val data = GeoPackage.load("file.gpkg")
 *     .toWGS84()
 *     .materialize()  // Cache transformed features for render loop
 * ```
 */
fun GeoSource.materialize(): GeoSource {
    val materializedFeatures = listFeatures()

    return object : GeoSource(crs) {
        override val features: Sequence<Feature> = materializedFeatures.asSequence()
    }
}
```

## Implementation Notes

### CRSTransformer Behavior
- **Fail fast:** Throw exception on invalid EPSG codes during construction, not during transformation
- **Lifecycle:** Caller owns CRSTransformer instance. No caching or shared state.
- **Thread safety:** Single-threaded transformations only. For thread-safe parallel transforms, create transformer per thread.

### Geometry.transform() Implementation
- **Immutability:** Always create new Geometry instances, don't modify in-place
- **Structural integrity:** Ensure all nested coordinates are transformed (e.g., Polygon holes)
- **Type safety:** Sealed class `when` ensures all geometry types are handled at compile time

### GeoSource.autoTransformTo() Behavior
- **Identity optimization:** Returns `this` if source CRS equals target CRS (same object, not copy)
- **Lazy evaluation:** Returns new GeoSource with lazy transformed Sequence
- **CRS update:** New GeoSource has `crs = targetCRS`, preserving transformed coordinates

### materialize() Implementation
- **Purpose:** Escape hatch for render loops where features are accessed repeatedly
- **Behavior:** Forces Sequence evaluation into List, returns new GeoSource backed by List
- **Use case:** Animation/render loops, not default usage

### Error Handling
- **Invalid CRS:** Throw `CRSTransformationException` with clear message about proj4j-epsg dependency
- **Unknown projection:** Propagate proj4j exceptions with context about which CRS failed
- **No graceful degradation:** If CRS transformation fails, fail immediately. Don't silently ignore.

## Testing Strategy

### Synthetic-First Testing with Known Value Verification

**Goal:** Verify transformation accuracy using published Ordnance Survey control points. Synthetic tests provide reproducible verification without external data file dependencies.

**Test data:** Use Ordnance Survey published control points with known BNG coordinates and WGS84 equivalents:
- London TQ 530 180 → lon ≈ -0.1, lat ≈ 51.5
- Edinburgh NT 267 734 → lon ≈ -3.2, lat ≈ 55.9

**Accuracy expectations:**
- Helmert transformation (proj4j default): ~3-5 meter accuracy for BNG (creative coding acceptable)
- OSTN15 transformation: <0.1m accuracy (requires external PROJ C library or Grid InQuestII)
- Use Helmert by default, document accuracy expectations per CRS

### CRSTransformerTest.kt
```kotlin
class CRSTransformerTest {
    @Test
    fun testLondonBNGToWGS84() {
        val transformer = CRSTransformer("EPSG:27700", "EPSG:4326")
        val easting = 530000.0  // TQ 530
        val northing = 180000.0  // TQ 180

        val wgsResult = transformer.transform(easting, northing)

        // Published Ordnance Survey control point
        // London TQ 530 180 → lon ≈ -0.1, lat ≈ 51.5
        assertTrue(abs(wgsResult.x + 0.1) < 0.001, "Longitude should be ≈ -0.1")
        assertTrue(abs(wgsResult.y - 51.5) < 0.001, "Latitude should be ≈ 51.5")
    }

    @Test
    fun testBNGRoundTrip() {
        val toWGS = CRSTransformer("EPSG:27700", "EPSG:4326")
        val toBNG = CRSTransformer("EPSG:4326", "EPSG:27700")

        val original = Vector2(530000.0, 180000.0)
        val wgs84 = toWGS.transform(original.x, original.y)
        val backToBNG = toBNG.transform(wgs84.x, wgs84.y)

        // Round-trip within ~5 meters (Helmert accuracy)
        val error = Math.sqrt(
            Math.pow(original.x - backToBNG.x, 2) +
            Math.pow(original.y - backToBNG.y, 2)
        )
        assertTrue(error < 5.0, "Round-trip error: $error meters")
    }

    @Test(expected = CRSTransformationException::class)
    fun testInvalidCRSThrows() {
        CRSTransformer("EPSG:99999", "EPSG:4326")
    }
}
```

## Open Questions

### 1. GeoJSON CRS Property (RFC 7946 vs 2008)
**What we know:** RFC 7946 removed `crs` member, assumes WGS84. Older GeoJSON 2008 files may have `crs` property.
**What's unclear:** Should we attempt to read obsolete `crs` member for backward compatibility?
**Recommendation:** Default to "EPSG:4326" for GeoJSON. Don't attempt to read obsolete `crs` property. Document that non-WGS84 GeoJSON files need manual CRS specification.

### 2. WKT2 and PROJ JSON Support
**What we know:** proj4j supports PROJ strings and EPSG codes. PROJ 9.7+ supports WKT2 and PROJ JSON.
**What's unclear:** Does proj4j 1.4.1 support WKT2 or PROJ JSON CRS definitions?
**Recommendation:** Start with EPSG codes only (epsg:XXXX format). If custom CRS needed, investigate WKT2 support in proj4j.

### 3. Caching Strategy for Repeated Transforms
**What we know:** CONTEXT.md says "no cache" for simplicity. User owns lifecycle.
**What's unclear:** Are there scenarios where caching CoordinateTransform objects provides value?
**Recommendation:** Follow CONTEXT.md decision: No cache. CRSTransformer creates transform once per instance.caller manages lifecycle.
