# Phase 4.1: CRS-Aware GeoSource with Auto-Reprojection - Context

**Gathered:** 2026-02-22
**Status:** Ready for planning

<domain>
## Phase Boundary

Automatically detect CRS metadata in loaded data and handle coordinate transformations, eliminating manual CRS management in user code. GeoSource should transform its own coordinates to match the rendering projection, removing the ~50 lines of boilerplate currently required for BNG data.

</domain>

<decisions>
## Implementation Decisions

### API design
- Provide both `toWGS84()` convenience method and explicit `autoTransformTo(targetCRS)` for custom CRS
- `toWGS84()` is the recommended first step after loading data, since all GeoProjection implementations expect WGS84 coordinates
- Returns `this` unchanged if source CRS already matches target CRS (identity check)

### Performance strategy
- Transform coordinates immediately when `autoTransformTo()` is called
- Default behavior produces lazy Sequence<Feature> (respects existing contract from Phase 1)
- Provide explicit `materialize()` escape hatch for render loops where features are iterated many times
- User chooses between lazy (good for single-pass, large datasets) and eager (good for animation/render loops)

### Error handling
- Fail fast with clear error if CRS is not available in proj4j when constructing CRSTransformer
- Throw exception on invalid EPSG codes during transformer construction
- No graceful degradation — unknown CRS is an immediate hard failure

### Caching strategy
- **No cache.** CRSTransformer is a simple wrapper around CoordinateTransform. Caller owns the instance lifecycle.
- Construction happens once in constructor (CRSFactory + CoordinateTransformFactory)
- Typical application has 2-3 distinct CRS pairs — no need for complex cache management
- YAGNI principle: cache would add thread-safety complexity without solving a real problem

### Sequence evaluation
- **Lazy by default.** `GeoSource.features` returns Sequence, preserving Phase 1's memory-efficient design
- `autoTransformTo()` returns new GeoSource with lazy transformed Sequence
- New `materialize()` method returns eager GeoSource backed by in-memory List
- Document tradeoff: lazy = per-frame CRS transform cost; eager = upfront cost + memory

### Extension placement
- New file: `src/main/kotlin/geo/projection/CRSExtensions.kt`
- Contains all CRS convenience extensions for GeoSource
- Separates CRS concerns from screen transform concerns (ScreenTransform.kt handles screen ↔ geo, CRSExtensions.kt handles CRS ↔ CRS)
- Maintains proper dependency flow: `geo.projection` depends on both proj4j and `geo.*` types

### Testing approach
- Synthetic-first with known value verification (published Ordnance Survey control points)
- `CRSTransformerTest.kt`: unit tests with exact survey points (London TQ 530 180 → lon ≈ -0.1, lat ≈ 51.5), round-trip tests, identity tests, batch tests
- `GeometryTransformTest.kt`: structural integrity tests verifying all geometry types (Point, LineString, Polygon, Multi*) transform correctly
- One integration test with real GeoPackage data in `GeoPackageTest.kt` or `CRSIntegrationTest.kt`
- Testing for expected accuracy: BNG Helmert transformation ~3-5m (intentional for creative coding)

### Documentation
- KDoc on `autoTransformTo()` and `toWGS84()` with usage examples
- Before/after migration example in `LiveRendering.kt` showing 40-line BNG boilerplate → 1-line `toWGS84()`
- "CRS Handling" section in README or `docs/crs-guide.md` covering:
  - When CRS transformation is needed (check `data.crs` after loading)
  - One-liner fix: `.toWGS84()` or `.autoTransformTo("EPSG:XXXX")`
  - Performance guidance: use `.materialize()` in render loops
  - Accuracy expectations per CRS
  - Unsupported CRS handling

### OpenCode's Discretion
- Exact progress bar implementation during transformation not specified
- How to document accuracy expectations for different CRS transformations
- Whether to add additional CRS convenience extensions beyond `toWGS84()` and `toWebMercator()`

</decisions>

<specifics>
## Specific Ideas

- Problem: LiveRendering.kt demonstrates the pain point — ~50 lines of manual CRS handling for BNG data (detect CRS, convert bounds, write helper function, call helper per geometry type, magic scale multiplier)
- Solution example: `val data = GeoPackage.load("data/geo/ness-vectors.gpkg").toWGS84().materialize()` — single line replacement
- Design principle: Make GeoSource responsible for presenting coordinates in canonical CRS (WGS84), so projections always receive consistent input
- Design principle: Tell, Don't Ask — instead of asking source for CRS and telling each coordinate to convert, tell the source to transform itself
- Design principle: Polymorphism over Conditionals — `Geometry.transform()` dispatched per type via sealed class hierarchy
- Use published Ordnance Survey control points for testing reproducibility
- Single responsibility: CRSTransformer owns coordinate math; Geometry.transform() owns structural traversal; GeoSource.autoTransformTo() owns the pipeline
- Document the cost/benefit tradeoff: lazy transformations re-run on every frame, materialize() caches upfront

</specifics>
